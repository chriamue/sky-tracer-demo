What is MCP?

Model Context Protocol (MCP) is an open protocol that enables seamless integration between LLM applications and external data sources and tools.

Similar to Language Server Protocol (LSP) for code editors, MCP standardizes how to integrate context and tools into the AI ecosystem.

Why MCP?

  • 🔗 Connect LLMs to any data source or tool
  • 📦 Standardized, vendor-neutral protocol
  • 🔄 Seamless integration with AI applications
  • 🛡️ Built-in security and consent mechanisms
  • ♻️ Composable and reusable integrations

Inspired by Language Server Protocol (LSP)

📝 Language Server Protocol

  • Standardizes language support
  • Works across many editors
  • One implementation → many tools
  • Examples: TypeScript, Python, Rust

🤖 Model Context Protocol

  • Standardizes AI tool integration
  • Works across many AI apps
  • One server → many AI models
  • Examples: Claude, LM Studio

Just as LSP eliminated the need for language support in every editor, MCP eliminates the need for integration in every AI application.

  • Client: Connector within the host

Server Features

📚 Resources

Context and data for the AI model to use

Files, documents, structured data

🛠️ Tools

Functions the AI can execute

API calls, data queries

💬 Prompts

Templated messages and workflows

Reusable message templates

How MCP Works

  1. Host establishes connection to MCP Server
  2. Server declares capabilities (resources, tools, prompts)
  3. Host/AI system can now use server's capabilities
  4. JSON-RPC 2.0 messages carry requests/responses
  5. Server can request LLM sampling with user consent

The Protocol: JSON-RPC 2.0

MCP messages are JSON-RPC 2.0 over a transport (HTTP/SSE, stdio):

Request


{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "list_flights",
    "arguments": {
      "departure": "FRA"
    }
  }
}

Response


{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "[{\"flight\": \"LH100\",
          \"dest\": \"JFK\"}]"
      }
    ]
  }
}

Protocol Lifecycle

sequenceDiagram
    participant Host as Host (LM Studio)
    participant Server as MCP Server (Sky Nexus)

    Note over Host,Server: 1. Initialization
    Host->>Server: initialize (client capabilities)
    Server-->>Host: server capabilities + tool list
    Host->>Server: notifications/initialized

    Note over Host,Server: 2. Discovery
    Host->>Server: tools/list
    Server-->>Host: [{name, description, inputSchema}]

    Note over Host,Server: 3. Tool Call
    Host->>Server: tools/call {name, arguments}
    Server-->>Host: {content: [{type, text}]}

    Note over Host,Server: 4. Shutdown
    Host->>Server: close connection

Communication Flow

sequenceDiagram
    participant AI as AI System
    participant Client as MCP Client
    participant Server as MCP Server
    participant Data as External Data

    AI->>Client: Use tool "list_flights"
    Client->>Server: JSON-RPC tools/call
    Server->>Data: Query flights
    Data-->>Server: Flight data
    Server-->>Client: Tool result
    Client-->>AI: Flight information

Security & Trust

  • ✅ User consent required for data access
  • ✅ User control over what data is shared
  • ✅ Tool execution requires explicit approval
  • ✅ LLM sampling requests need user approval
  • ✅ Clear documentation of security implications

Key Principle: Users must understand and consent to all data access and operations.

Sky Nexus: An MCP Server

Sky Nexus is a Model Context Protocol server that provides comprehensive aviation data services.

It acts as a central hub, integrating with multiple microservices and exposing aviation data through both REST API and MCP tools.

Features

  • 🔍 OpenAPI docs (Swagger UI)

MCP Endpoints Overview

  • 🏢 /mcp/airports - Airport lookup and information
  • 🎮 /mcp/flights - Flight management and search
  • 🛰️ /mcp/satellites - Satellite positioning and tracking
  • ⏰ /mcp/datetime - Timezone and time utilities
  • 🗼 /mcp/babel - Flight tracking and aggregation

Airport Tools


{
  "tools": [
    {
      "name": "list_airports",
      "description": "List all available airports",
      "inputSchema": {}
    },
    {
      "name": "get_airport",
      "description": "Get airport details by code",
      "inputSchema": {
        "type": "object",
        "properties": {
          "code": {"type": "string", "description": "IATA code"}
        }
      }
    }
  ]
}

Flight Tools

  • list_flights - List all flights with optional filters
  • get_flight - Get detailed flight information
  • create_flight - Create a new flight
  • search_flights_by_route - Search by departure and arrival

Satellite Tools

  • list_satellites - List all satellites
  • create_satellite - Create a new satellite
  • update_satellite_status - Update satellite status
  • calculate_position - Calculate flight position using satellites

DateTime Tools

  • get_current_datetime - Get time in any timezone with custom formatting
  • get_aviation_times - Get current time in major aviation hubs
  • get_timezone_difference - Calculate time difference between timezones

🌍 Supports all IANA timezone identifiers

Babel Tools

Flight tracking and aggregation across multiple services

  • track_flight - Track real-time flight position and status
  • search_flights - Search flights across the network
  • get_flight_aggregation - Get aggregated flight data

  1. Start docker compose: docker compose up -d
  2. Open LM Studio
  3. Program tab (right sidebar) → "Install > Edit mcp.json"

Example: Using Sky Nexus

Once configured in your frontend (Claude, LM Studio, etc.):

"Find all flights departing from Frankfurt (FRA) to Lisbon (LIS) in the next 24 hours"

The LLM will:

  1. Call search_flights_by_route tool
  2. Receive flight data from Sky Nexus
  3. Present results in natural language

Example: Multi-Timezone Flight Planning

"What time does flight SK123 depart from Copenhagen and what time will it arrive in Sydney? Show both in local times."

Sky Nexus tools handle:

  1. Get flight details (SK123)
  2. Retrieve airport timezone info
  3. Convert times using get_current_datetime
  4. Return localized flight times

Example: Real-time Satellite Tracking

"Where is flight BA286 right now? Use satellite data to calculate its position."

Workflow:

  1. Get flight info from Flight Controller
  2. Use calculate_position with satellite data
  3. Get departure/arrival from airports
  4. Present location with context
