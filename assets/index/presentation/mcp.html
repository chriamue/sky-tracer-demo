What is MCP?

Model Context Protocol (MCP) is an open protocol that enables seamless integration between LLM applications and external data sources and tools.

Similar to Language Server Protocol (LSP) for code editors, MCP standardizes how to integrate context and tools into the AI ecosystem.

Why MCP?

  • 🔗 Connect LLMs to any data source or tool
  • 📦 Standardized, vendor-neutral protocol
  • 🔄 Seamless integration with AI applications
  • 🛡️ Built-in security and consent mechanisms
  • ♻️ Composable and reusable integrations

Inspired by Language Server Protocol (LSP)

📝 Language Server Protocol

  • Standardizes language support
  • Works across many editors
  • One implementation → many tools
  • Examples: TypeScript, Python, Rust

🤖 Model Context Protocol

  • Standardizes AI tool integration
  • Works across many AI apps
  • One server → many AI models
  • Examples: Claude, LM Studio

Just as LSP eliminated the need for language support in every editor, MCP eliminates the need for integration in every AI application.

Three Primitives

🔧 Tools

Model decides when to call them

Execute actions, query data, mutate state — like function calls

list_flights()

📦 Resources

Client reads them by URI

Read-only data injected as context — like a typed file system for the model

airports://FRA

💬 Prompts

User invokes them by name

Pre-built conversation templates filled with live data — for structured workflows

airport-briefing

Each primitive has a different controller: the model, the client, or the user.

🔧 Tools — Model-Driven Actions

The model autonomously decides to call a tool based on the conversation. Tools have a typed JSON Schema input and return structured content.

Tool definition (server side)

{
  "name": "list_flights",
  "description": "List all flights with optional filters",
  "inputSchema": {
    "type": "object",
    "properties": {
      "departure": {
        "type": "string",
        "description": "Filter by departure airport"
      }
    }
  }
}

Invocation (JSON-RPC)

{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "list_flights",
    "arguments": {
      "departure": "FRA"
    }
  }
}

💡 The model sees the tool list after initialize and calls tools as needed — no user action required.

📦 Resources — URI-Addressed Data

Resources are read-only data identified by a URI. The client fetches them and injects their content as context — the model doesn't call them directly.

Resource request

{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "resources/read",
  "params": {
    "uri": "airports://FRA"
  }
}

URI template

{
  "uriTemplate": "airports://{code}",
  "name": "Airport by IATA code",
  "mimeType": "application/json"
}

Resource response

{
  "contents": [{
    "uri": "airports://FRA",
    "mimeType": "application/json",
    "text": "{
  \"code\": \"FRA\",
  \"name\": \"Frankfurt Airport\",
  \"latitude\": 50.033,
  \"longitude\": 8.570
}"
  }]
}

💡 Unlike tools, resources are passive — they are data, not actions.

💬 Prompts — User-Invoked Templates

Prompts are pre-built conversation starters with parameters. The server fills them with live data and returns a ready-to-send message sequence.

Get prompt request

{
  "method": "prompts/get",
  "params": {
    "name": "airport-briefing",
    "arguments": {
      "airport_code": "FRA"
    }
  }
}

Server injects live data and returns

{
  "messages": [
    {
      "role": "user",
      "content": {
        "type": "text",
        "text": "Provide a briefing for:\n
Airport: Frankfurt Airport (FRA)\nLatitude: 50.033\n..."
      }
    },
    {
      "role": "assistant",
      "content": { "type": "text",
        "text": "I'll brief you on FRA..." }
    }
  ]
}

💡 Prompts orchestrate complex workflows — they can call internal services to enrich the template before returning it.

How MCP Works

  1. Host establishes connection to MCP Server
  2. Server declares capabilities (resources, tools, prompts)
  3. Host/AI system can now use server's capabilities
  4. JSON-RPC 2.0 messages carry requests/responses
  5. Server can request LLM sampling with user consent

The Protocol: JSON-RPC 2.0

MCP uses JSON-RPC 2.0 over HTTP Streamable (POST + SSE) or stdio:

Tool call request

{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "list_flights",
    "arguments": { "departure": "FRA" }
  }
}

Resource read request

{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "resources/read",
  "params": { "uri": "airports://FRA" }
}

Tool call response

{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [{
      "type": "text",
      "text": "[{\"flight\":\"LH100\",\"dest\":\"JFK\"}]"
    }]
  }
}

Image content (map tool)

{
  "content": [{
    "type": "image",
    "mimeType": "image/png",
    "data": "iVBORw0KGgo..."
  }]
}

Protocol Lifecycle

sequenceDiagram
    participant Host as Host (LM Studio / Claude)
    participant Server as MCP Server (Sky Nexus)

    Note over Host,Server: 1. Initialization
    Host->>Server: initialize (client capabilities)
    Server-->>Host: capabilities: tools + resources + prompts
    Host->>Server: notifications/initialized

    Note over Host,Server: 2. Discovery
    Host->>Server: tools/list, resources/list, prompts/list
    Server-->>Host: tool schemas, resource URIs, prompt names

    Note over Host,Server: 3. Usage
    Host->>Server: tools/call {name, arguments}
    Server-->>Host: {content: [{type, text|image}]}
    Host->>Server: resources/read {uri: "airports://FRA"}
    Server-->>Host: {contents: [{uri, mimeType, text}]}
    Host->>Server: prompts/get {name, arguments}
    Server-->>Host: {messages: [{role, content}]}

Communication Flow

sequenceDiagram
    participant AI as AI System
    participant Client as MCP Client
    participant Server as MCP Server
    participant Data as External Data

    AI->>Client: Use tool "list_flights"
    Client->>Server: JSON-RPC tools/call
    Server->>Data: Query flights
    Data-->>Server: Flight data
    Server-->>Client: Tool result
    Client-->>AI: Flight information

Security & Trust

  • ✅ User consent required for data access
  • ✅ User control over what data is shared
  • ✅ Tool execution requires explicit approval
  • ✅ LLM sampling requests need user approval
  • ✅ Clear documentation of security implications

Key Principle: Users must understand and consent to all data access and operations.

Single MCP Endpoint

Sky Nexus exposes one MCP endpoint that advertises all capabilities:

http://localhost:8000/mcp

🔧 18 Tools

Airports, Flights, Satellites, DateTime, Tracking, Map

📦 8 Resources

airports://FRA and 7 other featured airports + URI template

💬 4 Prompts

airport-briefing, flight-route-analysis, delay-investigation, network-overview

Transport: HTTP Streamable (JSON-RPC 2.0 over POST + SSE)

Airport Tools


{
  "tools": [
    {
      "name": "list_airports",
      "description": "List all available airports",
      "inputSchema": {}
    },
    {
      "name": "get_airport",
      "description": "Get airport details by code",
      "inputSchema": {
        "type": "object",
        "properties": {
          "code": {"type": "string", "description": "IATA code"}
        }
      }
    }
  ]
}

Flight Tools

  • list_flights - List all flights with optional filters
  • get_flight - Get detailed flight information
  • create_flight - Create a new flight
  • search_flights_by_route - Search by departure and arrival

Satellite Tools

  • list_satellites - List all satellites
  • create_satellite - Create a new satellite
  • update_satellite_status - Update satellite status
  • calculate_position - Calculate flight position using satellites

DateTime Tools

  • get_current_datetime - Get time in any timezone with custom formatting
  • get_aviation_times - Get current time in major aviation hubs
  • get_timezone_difference - Calculate time difference between timezones

🌍 Supports all IANA timezone identifiers

Live Tracking Tools

Real-time flight tracking via the Tower of Babel aggregation service

  • get_flights_by_airport - Departures & arrivals at an airport
  • get_flight_position - Current lat/lon of a specific flight
  • search_flights_by_airport_pattern - Pattern-based airport search

🗺️ Map Tool

Generates a world map image showing active routes and airport pins

  • generate_flight_map — renders a PNG world map

Parameters:

  • airport_code (optional) — focus on one airport's routes
  • title (optional) — map title overlay

Returns a base64-encoded PNG image via Content::image("image/png")

Built with: Natural Earth GeoJSON → geo-svg → resvg → PNG

📦 Resources in Sky Nexus

Airport data is exposed as URI-addressed resources using the airports:// scheme

airports://FRA airports://LHR airports://CDG airports://JFK airports://DXB airports://SIN airports://AMS airports://HND

URI template: airports://{code} — any IATA code works

Each resource returns application/json with live data:

{ "code": "FRA", "name": "Frankfurt Airport", "latitude": 50.033, "longitude": 8.570 }

Resources let clients pre-load airport context before asking the model to reason about it — without a tool call.

💬 Prompts in Sky Nexus

4 workflow templates — each fetches live data and returns a pre-filled conversation

  • airport-briefing airport_code
    Full airport briefing: location, weather, airlines, terminals, ops notes
  • flight-route-analysis departure, arrival, date?
    Route distance, duration, carriers, time zones, overflight countries
  • delay-investigation airport_code
    Injects live flight data → structured delay report with root-cause analysis
  • aviation-network-overview (no params)
    Lists all tools, resources, prompts and suggests example queries

MCP Integration Options

Sky Nexus can integrate with multiple AI frontends:

🎯 Claude Desktop

Official MCP support via config file

MCP Host with mcp.json configuration

Claude Desktop Configuration

Start docker compose, then add to Claude Desktop config:


{
  "mcpServers": {
    "sky-nexus": {
      "url": "http://localhost:8000/mcp"
    }
  }
}

One server — all 18 tools, 8 resources, and 4 prompts auto-discovered via initialize

LM Studio Configuration

Configure Sky Nexus via LM Studio:

  1. Start docker compose: docker compose up -d
  2. Open LM Studio → Program tab (right sidebar)
  3. "Install → Edit mcp.json"
  4. Add Sky Nexus server

{
  "mcpServers": {
    "sky-nexus": {
      "url": "http://localhost:8000/mcp"
    }
  }
}

⚠️ LM Studio currently exposes tools only — resources and prompts require MCP Inspector or Claude Desktop

LM Studio as MCP Host

  • 💻 Local frontend for any LLM
  • 🔌 Connects MCP servers to the app
  • 🛡️ Manages all MCP tool access
  • ⚙️ Configuration via mcp.json
  • 🔐 Supports authentication headers if needed

⚠️ Only install MCPs from trusted sources - they can execute code and access local files

Testing with MCP Inspector

Browser UI to explore tools, resources and prompts interactively:


# Start docker compose first
docker compose up -d

# Launch MCP Inspector against the single endpoint
npx @modelcontextprotocol/inspector \
  --transport http \
  --server-url http://localhost:8000/mcp

The Inspector lets you: browse the tool list, call tools with custom args, read resources by URI, and invoke prompts — all without writing any code.

Example: Using Sky Nexus

Once configured in your frontend (Claude, LM Studio, etc.):

"Find all flights departing from Frankfurt (FRA) to Lisbon (LIS) in the next 24 hours"

The LLM will:

  1. Call search_flights_by_route tool
  2. Receive flight data from Sky Nexus
  3. Present results in natural language

Example: Multi-Timezone Flight Planning

"What time does flight SK123 depart from Copenhagen and what time will it arrive in Sydney? Show both in local times."

Sky Nexus tools handle:

  1. Get flight details (SK123)
  2. Retrieve airport timezone info
  3. Convert times using get_current_datetime
  4. Return localized flight times

Example: Real-time Satellite Tracking

"Where is flight BA286 right now? Use satellite data to calculate its position."

Workflow:

  1. Get flight info from Flight Controller
  2. Use calculate_position with satellite data
  3. Get departure/arrival from airports
  4. Present location with context
